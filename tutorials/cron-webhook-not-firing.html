Cron Webhook Not Firing?

If your scheduled job runs but your webhook destination gets nothing, this guide isolates the issue in minutes without guesswork.

Fast reality check

Most webhook failures are one of these:

  • Job ran, destination rejected it (4xx/5xx response).
  • Wrong URL (staging vs production, typo, old endpoint).
  • Delivery mode mismatch (announce vs webhook).
  • Webhook endpoint requires auth/signature that the job did not send.

2-minute diagnosis flow

  1. Prove delivery mode.
    Ask for the full job object and confirm it has delivery.mode: webhook and delivery.to: https://....
  2. Run a canary job now.
    Trigger a one-shot job with unique payload text like CANARY-WEBHOOK-APR03.
  3. Check run history.
    Confirm whether the cron run finished and capture any error status/details.
  4. Verify receiver logs.
    Check your webhook endpoint logs for request arrival, status code, and rejection reason.

If run history shows success but receiver has no hit, your URL/destination is likely wrong. If receiver shows 4xx/5xx, fix endpoint auth/format first.

High-friction mistakes

  • Using localhost/private URL: remote jobs cannot reach http://localhost on your laptop.
  • Trailing slash/path mismatch: /webhook vs /webhook/ can be treated differently.
  • HTTP instead of HTTPS: some environments reject insecure endpoints.
  • Wrong method expectation: endpoint expects GET, cron sends POST.
  • Missing auth header/secret: endpoint drops request as unauthorized.
  • “It ran” interpreted as “it delivered”: execution success and delivery success are different checks.

Advanced edge cases (when basics look correct)

  • 302/307 redirect trap: your URL redirects to login or another host, so receiver logs look empty on the original endpoint.
  • TLS/certificate mismatch: endpoint cert is expired/self-signed or hostname does not match; request fails before app logs.
  • Provider WAF/rate limits: Cloudflare/API gateways return intermittent 403/429 during bursts.
  • Idempotency dedupe: receiver silently drops repeat payload IDs/messages as duplicates.
  • Wrong environment secret: staging secret checked against production endpoint (or vice versa).

If delivery fails intermittently, compare two back-to-back canaries with unique payload IDs and inspect both receiver and edge/proxy logs.

Copy/paste recovery prompt

Show the full cron job JSON for my webhook task. Verify delivery.mode is webhook and echo delivery.to exactly. Then create a one-shot canary job 2 minutes from now with payload text CANARY-WEBHOOK-DELIVERY-TEST and delivery.to set to the same URL. After it runs, return run status + any delivery error details verbatim.

When to escalate

Escalate with this packet:

  • Job ID + full delivery block
  • Canary payload text and trigger time (local + UTC)
  • Run-history result from scheduler
  • Receiver-side HTTP status + error/body snippet
  • Any redirect/TLS/proxy layer evidence (Cloudflare, API Gateway, reverse proxy logs)

Without scheduler + receiver evidence together, support can’t tell if the failure is routing, auth, or endpoint behavior.

Bottom line: verify mode, URL, run history, and receiver logs in one pass. That closes almost every “webhook didn’t fire” case fast.
